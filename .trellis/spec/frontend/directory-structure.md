## Rules

### 1. Route files stay thin

`src/app/page.tsx` should assemble static page data and render the screen entry component. Do not turn route files into giant UI files.

### 2. Group interactive UI by screen or domain

Interactive workbench modules live in `src/components/dashboard/`.

Current examples:

- `study-wall-dashboard.tsx` for the main learning workbench.
- `signal-algorithm-lab.tsx` for signal-and-system algorithm demonstrations.

### 3. Keep reusable UI and system wrappers focused

- Markdown/LaTeX rendering belongs in `src/components/ui/math-markdown.tsx`.
- Small visual wrappers belong in `src/components/ui/`.
- Browser registration wrappers such as PWA setup belong in `src/components/system/`.

### 4. Keep domain contracts separate from seed data

- Types go to `src/types/`.
- Seed data and import adapters go to `src/lib/`.
- Do not widen UI contracts to match raw research payloads.

### 5. Research code never mixes with UI code

Anything that touches target sites for analysis belongs in `scripts/`, `research/`, or `docs/research/`, not in runtime UI modules.

### 6. Static export artifacts are generated, not authored

GitHub Pages output is generated by `next build` and synced with `npm run sync:pages`. Do not hand-edit root-level exported HTML, `_next/`, or `_not-found/`.

## Anti-patterns

- Putting scraping logic under `src/`.
- Putting all UI into `src/app/page.tsx`.
- Reusing target-site raw payload shapes directly in components.
- Manually editing static export output instead of changing source files.

# Study Wiki Wall 前端架构基线

> 更新时间：2026-04-21

## 1. 结论

主应用采用：

- **Next.js App Router**
- **TypeScript**
- **Tailwind CSS**
- **react-markdown + KaTeX 数学渲染链路**

暂不将 Astro 作为主架构。

## 2. 原因

### 2.1 项目本质是 Web App，不是内容站

当前核心能力更接近：

- 学习记录
- 题目筛选与切换
- 用户态交互
- 评论 / 资源 / 统计
- 多学科、多试卷、多题型切面

这类产品的重点是：

- 路由与数据边界
- 服务端 / 客户端混合渲染
- 复杂交互状态组织
- 后续和后端 API 的稳定衔接

Next.js 更适合做主应用骨架。

### 2.2 Astro 更适合外围内容层

Astro 更适合：

- 官网
- 博客
- 帮助中心
- 文档站
- 落地页

如果后续需要营销官网，可以把 Astro 当作外围站点方案，而不是当前主产品架构。

## 3. 当前前端分层

### 路由层

- `src/app/layout.tsx`：根布局、metadata、全局样式
- `src/app/page.tsx`：首页 server component，负责拼装页面数据与客户端入口

### 交互编排层

- `src/components/dashboard/study-wall-dashboard.tsx`
- `src/hooks/use-study-wall-dashboard.ts`

说明：

- 复杂筛选、搜索、题目可见集计算，集中在一个屏幕级 client module 中
- 当前默认优先进入 `signal-system`，方便继续做信号与系统内容导入
- 单题 / 概览双视图逻辑也集中在 dashboard hook 中组织

### 渲染增强层

- `src/components/ui/math-markdown.tsx`

说明：

- 题目内容当前统一按 Markdown 文本处理
- 若字符串中包含数学公式，则通过 `remark-math + rehype-katex` 渲染
- 这样可兼容普通文本、列表、行内公式和块公式

### 导图布局层

- `src/components/dashboard/study-wall-dashboard.tsx` 内部的导图构建与布局函数

说明：

- 当前导图不是通用白板，而是面向 `科目 -> 章节 -> 知识点 -> 题目` 的学习树布局
- 2026-04-27 起，布局不再依赖题目内容渲染档位，而是改为：
  - 数据树构建
  - 递归测量子树权重
  - 基于子树密度的动态纵横间距
  - 独立的连线路由
  - 独立的 LOD 内容降级
- 进一步新增两层底层优化：
  - 子树签名复用：章节 / 知识点折叠与题目详情展开时，未变化的子树节点直接复用上次布局结果，减少整棵树重复测量
  - 缩放快照层：缩放进行中，题目节点优先显示轻量快照层，缩放停下后再恢复真实内容层
- 这样可以避免缩放时因为 LaTeX / 文本切档触发整棵树重排
- 题目卡缩小时使用遮罩占位，中档使用纯文本摘要，高档才渲染完整数学预览
- 当前策略更接近“学习导图”而非自由摆放式无限白板，后续若要升级为更通用脑图引擎，应继续拆出独立 `mindmap layout engine`

### 领域模型层

- `src/types/study-wall.ts`
- `src/lib/mock-study-wall.ts`

说明：

- UI 使用的是我们自己的领域模型，而不是直接照搬目标站点字段名
- 目前 `mock-study-wall.ts` 同时承担“前端种子数据”与“导入结构验证”的职责
- 已经验证 `科目 -> 章节 -> 知识点 -> 题型 -> 题目` 这一层级对当前场景是可用的

### 研究层

- `docs/research/`
- `scripts/fetch-target-site-snapshot.mjs`

说明：竞品研究与产品实现分离，避免把爬取逻辑带进页面组件。

## 4. 当前已验证的产品结构

当前前端已经验证了以下结构在学习墙场景中成立：

1. 顶部独立导航栏
2. 左侧章节/知识点可展开树
3. 中间主工作台负责显示概览或单题
4. 右侧只在选中单题时显示题目上下文
5. 掌握状态在题目层维护，而不是挂在章节层
6. 数学内容必须支持 LaTeX，否则信号与系统/数学题目可读性会明显下降

## 5. clean-room 原则

1. **研究结果可以保留**：页面结构、模块职责、交互流程、公开接口类型
2. **不要直接复用对方代码**
3. **不要让页面组件依赖竞品原始字段**
4. **先抽象自己的领域模型，再做 UI**
5. **把抓取脚本与产品代码隔离**
6. **文档导入内容允许做结构化整理，但不要标成原始权威文本**
